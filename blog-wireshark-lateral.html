Wireshark Deep Dive:
Detecting Lateral Movement

Mar 1, 2025 Wireshark Network Forensics Blue Team ~18 min read

Lateral movement is what separates a contained breach from a full network compromise. Attackers who've gained an initial foothold use techniques like Pass-the-Hash, Kerberoasting, PsExec, and WMI to spread through the network. This guide shows exactly how to spot these in Wireshark packet captures using targeted display filters and Tshark one-liners.

Setting Up Your Analysis Environment

Before diving into filters, set up Wireshark for efficient lateral movement analysis:

Wireshark Profile Setup
# Enable these columns:
# Time, Source, Destination, Protocol, Length, Info

# Key preferences:
# Edit > Preferences > Name Resolution
# - Enable "Resolve network (IP) addresses"
# - Enable "Resolve transport names"

# Color rules for quick triage:
# Red:    tcp.flags.reset==1 (RST flood = scanning)
# Orange: smb || smb2       (SMB traffic)
# Purple: kerberos          (Kerberos auth)
# Blue:   ntlmssp           (NTLM auth)

Detecting Pass-the-Hash

Pass-the-Hash (PtH) allows attackers to authenticate using just the NTLM hash — no plaintext password needed. The telltale sign is NTLM authentication from unexpected sources:

Wireshark Display Filter — PtH Detection
# Spot NTLM authentication attempts
ntlmssp.auth.username

# NTLM over SMB — most common PtH vector
smb2 && ntlmssp

# Failed NTLM auths in bulk = spray/PtH attempt
ntlmssp.auth && (smb2.nt_status == 0xC000006D ||
                 smb2.nt_status == 0xC0000064)

# SMB auth from non-standard sources (not DC)
smb2 && ntlmssp.auth.username && 
  !(ip.src == 192.168.1.10)  # substitute your DC IP
ℹ What to Look For

Successful NTLM auths from workstations to other workstations (lateral) rather than workstation → DC (normal). Also watch for the same hash authenticating from multiple sources within a short timeframe.

Detecting Kerberoasting

Kerberoasting requests service tickets (TGS) for service accounts, then cracks them offline. Look for unusual TGS-REQ patterns:

Wireshark — Kerberoasting Filters
# All Kerberos TGS requests
kerberos.msg_type == 12  # TGS-REQ

# RC4 encryption in TGS (weak, crackable — red flag)
kerberos.msg_type == 12 && kerberos.etype == 23

# Multiple TGS-REQ from same source in short time
# Use Statistics > Conversations after filtering kerberos
# Look for one IP making 10+ TGS requests rapidly

# Full Kerberoasting detection filter
kerberos && kerberos.msg_type == 12 && 
  kerberos.etype == 23 &&
  kerberos.req_body_cname
Tshark — Extract Kerberoasting Indicators
tshark -r capture.pcap \ -Y "kerberos.msg_type==12 && kerberos.etype==23" \ -T fields \ -e ip.src \ -e ip.dst \ -e kerberos.req_body_cname \ -e frame.time \ | sort | uniq -c | sort -rn

Detecting PsExec

PsExec creates a temporary service on the remote host via SMB. The named pipe and service creation pattern is distinctive:

Wireshark — PsExec Detection
# PsExec creates PSEXESVC service via SMB pipe smb2.filename contains "PSEXESVC" # More generic — service creation over SMB named pipe smb2.filename contains "svcctl" # Service manager calls over SMB (DCE/RPC) dcerpc && smb2 && (dcerpc.cmd == 15 || dcerpc.cmd == 12) # CreateService, StartService # Full PsExec chain filter smb2 && (smb2.filename contains "PSEXESVC" || smb2.filename contains "svcctl" || smb2.filename contains "admin$")

Detecting WMI-Based Movement

WMI (Windows Management Instrumentation) is a popular living-off-the-land technique. It uses DCOM over TCP port 135 + a dynamic high port:

Wireshark — WMI / DCOM Detection
# DCOM initial connection (port 135) tcp.port == 135 && dcerpc # WMI over DCOM — process creation (most dangerous) # Look for DCOM traffic followed by dynamic port (>1024) from same pair dcerpc && ip.dst == [target_ip] # Tshark — find DCOM conversations tshark -r capture.pcap \ -Y "tcp.port==135 && dcerpc" \ -T fields -e ip.src -e ip.dst \ | sort | uniq -c | sort -rn # impacket wmiexec leaves this pattern: # Source → 135/tcp (DCOM bind) # Source → [dynamic port] (actual WMI call) # 2 connections from same src within 1 second

Tshark One-Liners for Rapid Triage

Production Tshark Commands
# Top talkers by byte count tshark -r capture.pcap -q -z conv,ip | head -20 # All SMB tree connects (reveals share access) tshark -r capture.pcap -Y smb2.cmd==3 \ -T fields -e ip.src -e smb2.tree | sort | uniq -c # Failed auth attempts (spray detection) tshark -r capture.pcap \ -Y "ntlmssp.auth && smb2.nt_status==0xC000006D" \ -T fields -e ip.src -e ntlmssp.auth.username \ | sort | uniq -c | sort -rn # Detect port scanning (RST flood) tshark -r capture.pcap -Y "tcp.flags.reset==1" \ -T fields -e ip.src -e ip.dst -e tcp.dstport \ | awk '{print $1}' | sort | uniq -c | sort -rn | head # Extract all DNS queries tshark -r capture.pcap -Y dns.qry.name \ -T fields -e dns.qry.name | sort | uniq # Find cleartext credentials (basic auth, FTP, etc.) tshark -r capture.pcap \ -Y "http.authorization || ftp.request.command==PASS" \ -T fields -e ip.src -e http.authorization -e ftp.request.arg

Building a Lateral Movement Timeline

Once you've identified suspicious traffic, reconstruct the attacker's timeline:

Timeline Reconstruction
# Export to CSV for timeline analysis tshark -r capture.pcap \ -Y "smb2 || kerberos || ntlmssp || dcerpc" \ -T fields \ -e frame.time_epoch \ -e ip.src -e ip.dst \ -e _ws.col.Protocol \ -e _ws.col.Info \ -E separator=, -E quote=d \ > lateral_timeline.csv # Then in Python: import pandas as pd df = pd.read_csv('lateral_timeline.csv', names=['time','src','dst','proto','info']) df['time'] = pd.to_datetime(df['time'], unit='s') df.sort_values('time').to_csv('sorted_timeline.csv')
✓ Pro Tip

Always correlate packet timestamps with Windows Event Log entries (Event IDs 4624, 4625, 4648, 7045) for full lateral movement attribution. Packet captures give the what; event logs give the who and how.

Key Takeaways

  • PtH: Look for NTLM auths from workstation-to-workstation, especially with RC4
  • Kerberoasting: Bulk TGS-REQ with etype=23 (RC4) from one source = attacker collecting hashes
  • PsExec: The PSEXESVC string in SMB file names is the dead giveaway
  • WMI: DCOM on 135 followed immediately by dynamic port communication = remote process execution
  • Always sort by conversation frequency — attackers generate unusual traffic volumes
