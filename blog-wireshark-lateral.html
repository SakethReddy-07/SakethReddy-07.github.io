Wireshark Deep Dive: Detecting Lateral Movement

Mar 1, 2025 Wireshark PCAP Analysis Kerberoasting ~14 min read

Once an attacker establishes a foothold, lateral movement is how they spread through a network to reach high-value targets. Most of this movement generates distinctive network patterns that are invisible to endpoint tools but clearly visible in packet captures. This post covers how to spot the most common lateral movement techniques in Wireshark.

Setting Up Your Wireshark Environment

Before diving into detection, configure Wireshark for efficient threat hunting:

Key display filter shortcuts to know
-- Filter by protocol:
smb2          -- SMB v2 traffic
kerberos      -- Kerberos auth packets
ntlmssp       -- NTLM authentication
ldap          -- LDAP queries

-- Combine filters:
smb2 && ip.src == 192.168.1.50
kerberos && kerberos.msg_type == 30   -- TGS-REQ
ⓘ Pro Tip

Use Statistics → Conversations to quickly spot hosts making an unusual number of connections. A single host connecting to 20+ internal IPs in seconds is almost always scanning or moving laterally.

Detecting Pass-the-Hash (PtH)

Pass-the-Hash attacks use stolen NTLM hashes to authenticate without knowing the plaintext password. The network signature is NTLM authentication where the username appears on an unusual source host.

What to Look For

Wireshark display filter
ntlmssp.auth.username && smb2

-- Drill into NTLMSSP packets and check:
-- NTLMSSP_AUTH → NtChallengeResponse (length 24 = LM hash, longer = NT hash)
-- Workstation name doesn't match the source IP hostname
⚠ Red Flag

NTLM authentication from a workstation to multiple servers in rapid succession, especially to ADMIN$ or C$ shares, is a strong PtH indicator. Legitimate users don't normally authenticate to 10 machines in 30 seconds.

SMB Tree Connect to Admin Shares

Filter for admin share access
smb2.tree contains "ADMIN$" or smb2.tree contains "C$" or smb2.tree contains "IPC$"

-- Legitimate use: IT admins, some backup software
-- Suspicious: any non-admin workstation, or rapid sequential connections

Detecting Kerberoasting

Kerberoasting requests Kerberos service tickets (TGS) for service accounts, then cracks them offline. The attack generates TGS-REQ packets for SPNs that the attacker normally wouldn't access.

TGS-REQ Flood — The Key Signature

Kerberoasting detection filter
kerberos.msg_type == 12   -- TGS-REP (service ticket granted)

-- Look for:
-- Many TGS-REQ packets from single host in short time
-- Requests for unusual SPNs (MSSQLSvc, HTTP/*, CIFS/*)
-- RC4 encryption type (etype 23) — weaker, easier to crack
-- AES not used (attacker requests RC4 specifically)
Filter for RC4 downgrade (Kerberoasting indicator)
kerberos.etype == 23 && kerberos.msg_type == 12
-- etype 23 = RC4-HMAC — attacker forced downgrade from AES
-- etype 18 = AES256 — normal, not being Kerberoasted
⚠ Why RC4 Matters

Attackers specifically request RC4-encrypted tickets because they crack 10x faster than AES. A host requesting multiple RC4 service tickets for different SPNs in under a minute is almost certainly Kerberoasting.

Detecting WMI-Based Lateral Movement

Windows Management Instrumentation (WMI) is heavily abused for remote code execution. It uses DCOM over dynamic ports, making it harder to filter than SMB.

WMI / DCOM filter
-- DCOM traffic (WMI uses this)
dcerpc && tcp.dstport == 135

-- After initial DCOM negotiation, look for dynamic high ports:
dcerpc && tcp.dstport > 49152

-- WMI exec signature: IWbemServices::ExecMethod calls
-- Look for: "Win32_Process" and "Create" in DCERPC payloads
Tshark extraction — WMI method calls
tshark -r capture.pcap -Y "dcerpc" \
  -T fields -e ip.src -e ip.dst -e dcerpc.opnum \
  | sort | uniq -c | sort -rn

Detecting Mimikatz / LSASS Dumping via Network

While LSASS dumping is a local operation, the credentials harvested are immediately used for lateral movement. Look for authentication bursts following unusual process activity on a host:

Post-credential-dump network pattern
-- Within seconds/minutes of LSASS dump:
-- Sudden burst of SMB/Kerberos auth from one host
-- Auth attempts to DC (88/tcp Kerberos) in rapid succession
-- New hosts being accessed that the workstation never touched before

-- Filter: all auth from a suspect host in a time window
(smb2 || kerberos || ntlmssp) && ip.src == 192.168.1.100

Detecting PsExec Remote Execution

PsExec is one of the most common lateral movement tools. It creates a named pipe over SMB and drops a service binary on the remote host.

PsExec signature in Wireshark
-- Step 1: SMB connect to ADMIN$ share
smb2.tree contains "ADMIN$"

-- Step 2: Named pipe creation — PSEXESVC
smb2.filename contains "PSEXESVC"

-- Step 3: Service control manager activity over SVCCTL pipe
smb2.filename contains "svcctl"

-- Combined filter:
smb2 && (smb2.filename contains "PSEXESVC" || smb2.filename contains "svcctl")
✓ Detection Summary

PsExec always leaves a distinct trail: ADMIN$ tree connect → PSEXESVC binary write → svcctl pipe → service start. This 4-step sequence in Wireshark is near-definitive proof of PsExec lateral movement.

Tshark One-Liners for PCAP Forensics

When working with large PCAPs, Wireshark's GUI is too slow. Tshark on the command line is essential:

Tshark forensic one-liners
# Top talkers
tshark -r capture.pcap -q -z conv,ip | sort -k3 -rn | head -20

# All SMB filenames accessed
tshark -r capture.pcap -Y smb2 -T fields -e smb2.filename | sort -u

# All Kerberos service ticket requests + SPN
tshark -r capture.pcap -Y "kerberos.msg_type==12" \
  -T fields -e ip.src -e kerberos.SNameString

# All NTLM usernames seen
tshark -r capture.pcap -Y ntlmssp \
  -T fields -e ip.src -e ntlmssp.auth.username | sort -u

# DNS queries (C2 beaconing detection)
tshark -r capture.pcap -Y dns -T fields \
  -e ip.src -e dns.qry.name | sort | uniq -c | sort -rn

Key Takeaways

  • Lateral movement leaves network fingerprints — each technique has a distinct packet signature
  • Pass-the-Hash: NTLM auth from unexpected hosts to ADMIN$ shares in rapid bursts
  • Kerberoasting: many TGS-REQ with RC4 (etype 23) from a single host in seconds
  • PsExec: always creates PSEXESVC named pipe over SMB — unmistakable in PCAP
  • Use Tshark for large PCAPs — GUI is too slow; scripted analysis finds patterns faster
  • Statistics → Conversations is your first stop — outliers stand out immediately
